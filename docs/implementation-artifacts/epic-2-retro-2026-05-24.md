## Successes

1. **Infrastructure front-loading in story 2.1 eliminated friction for subsequent stories.** Jest config, ESM barrel conventions, workspace dependency wiring, `tsconfig.spec.json`, `no-restricted-imports` — all discovery cost paid once. Stories 2.2–2.4 had zero debug log entries or infrastructure surprises.

2. **Adversarial review caught real bugs in every story.** OOB silent-corruption in `setCell` (2.1), AC grid-size mismatch (2.2), inconsistent test rigor (2.2), missing density validation (2.4). The structured BMAD code-review workflow consistently found issues that the dev agent's self-review missed.

3. **Full follow-through on Epic 1 retro commitments.** Both action items completed (adversarial reviews on every story, story completion sequence). Two of three tech debt items resolved (`no-restricted-imports` landed in 2.1, `epic-1` status updated). Only the informational Tailwind deprecation note remains (low priority).

4. **Story specs were detailed enough to eliminate implementation ambiguity.** Hand-verified test fixtures, pseudocode, file structure targets, debug learnings from prior stories, and architecture references — all included in dev notes. This enabled the dev agent to focus on correctness rather than interpretation.

5. **Zero new technical debt.** Epic 2 was a net debt reducer — resolved 2 items from Epic 1 and created none.

---

## Challenges

1. **Dev agent self-review consistently missed invariant tests.** All 4 stories had findings where architectural contracts (immutability, fresh buffer allocation, input validation) were implemented correctly but not asserted by tests. The self-review answered "does it work?" but not "is every stated contract encoded as a test?"

2. **Missing validation at function boundaries.** 2 of 4 stories shipped without proper input validation during initial implementation: `setCell` had no OOB check and `createGrid` accepted non-integer dimensions (2.1); `randomizeGrid` had no density range validation (2.4). All fixed after adversarial review.

3. **Inconsistent test rigor within the same file.** In story 2.2, the blinker test spot-checked 5 of 25 cells while the glider test exhaustively verified all 100 cells. Same file, different standards. Standardized to exhaustive verification after review.

---

## Key Insights

1. **"Does it work?" and "is every invariant tested?" are different review questions.** The dev agent's self-review only asks the first. Adding explicit invariant-test verification to the self-review checklist addresses this systemically.

2. **The adversarial review is doing double duty.** It catches both design-level issues and missing invariant tests. Shifting invariant coverage earlier (via self-review) would let the adversarial review focus on harder problems — design, performance, architectural alignment.

3. **Front-loading infrastructure pays compound dividends.** Story 2.1 was the heaviest (17 files changed). Stories 2.2–2.4 were progressively frictionless. Apply the same pattern in Epic 3 — story 3.1 should establish React/Canvas/testing conventions for the rest of the epic.

4. **Retro commitments work when tracked.** 100% follow-through on Epic 1 action items across all 4 stories. The discipline of reviewing prior retro commitments at the start of each epic creates real accountability.
