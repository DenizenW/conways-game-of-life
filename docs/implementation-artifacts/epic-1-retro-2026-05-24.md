## Successes

1. **First-commit purity achieved cleanly.** The temp-dir + copy approach for overlaying Nx scaffold over an existing repo with 9 planning commits was elegant — no history rewriting, no orphan branches, no force pushes.

2. **Pre-tagging generators reduced downstream work.** Using `--tags=scope:*` flags on generators in story 1.1b meant 5 of 8 projects arrived with correct tags. Story 1.2 only needed to manually tag 3 projects.

3. **CI job pattern established and replicated.** Story 1.3 defined the setup pattern (checkout, pnpm, node, install, `nx affected`). Stories 1.4 and 1.5 replicated it with near-zero friction.

4. **Boundary violation dual-demonstration exceeded expectations.** Showing both that workspace imports fire the rule AND that npm imports don't was more honest than the AC required. Adversarial review in story 1.2 drove this improvement.

5. **Sprint-status sequencing pattern discovered and codified.** Story 1.1 paid the discovery cost; every subsequent story followed the established sequence.

---

## Challenges

1. **Nx 22 CLI drift from architecture docs.** The architecture was authored against Nx 18+ conventions. Nx 22.7.2 changed several behaviors: `--interactive=false` didn't suppress all prompts, `--frontendProject` required full scoped project name, NestJS app landed at workspace root (`api/`) not `apps/api/`, `@nx/next:setup-tailwind` is deprecated. Affected 3 of 7 stories (1.1, 1.1b, 1.2).

2. **Module boundary rule scope misunderstanding.** The epic AC specified `import * as React from 'react'` for the boundary demo, but `@nx/enforce-module-boundaries` only governs workspace project imports. Caught by the adversarial review in story 1.2 (1 HIGH finding). Corrected to use a workspace import (`@conways-game-of-life/api-client`).

3. **Epic status housekeeping.** `epic-1` status in `sprint-status.yaml` remains `in-progress` instead of `done`. Trivial fix but needs to happen before Epic 2 starts.

---

## Key Insights

1. **Adversarial reviews are essential safety nets.** The boundary-rule scope issue was missed by architecture, epics, and the dev agent — only caught by the adversarial review. This validates the practice for every story.

2. **Architecture docs written against older tool versions will drift.** Runtime flag verification (`--help`) before running generators prevents surprises. This should be a standard step in any generator-based story.

3. **Process patterns discovered early propagate well.** The sprint-status sequencing pattern and the CI job setup pattern were both established in early stories and consistently followed. Discovery cost is paid once; benefits compound.
