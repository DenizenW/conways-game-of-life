## Successes

1. **Story scoping and sequencing were excellent.** Each story mapped to exactly one PR's worth of work with unambiguous acceptance criteria. Sequencing was right: 3.1 laid the foundation, 3.2 added interaction, 3.3 added controls, 3.4 added utility actions, 3.5 added the slider. Each story extended the previous one without rework.

2. **Front-loading infrastructure in 3.1 paid compound dividends.** Story 3.1 absorbed all infrastructure cost — canvas setup, reducer pattern, simulation loop, responsive layout, test infrastructure (`jest-canvas-mock`, ResizeObserver polyfill, `@testing-library/jest-dom`). Stories 3.2–3.5 had zero debug log entries. This validates the same pattern from Epics 1 and 2, making it a three-epic proven strategy.

3. **The `useSimulationLoop` hook built to production spec in 3.1 needed zero changes when 3.5 arrived.** The architecture's forward-thinking design (reading `genPerSec` via `useRef` instead of a `useEffect` dependency) meant story 3.5 only had to feed dynamic state instead of a constant. The hook was untouched. Architecture paying dividends.

4. **69 tests, 0 flakes, 0 production incidents.** Test infrastructure established in 3.1 worked reliably across all 5 stories. No tooling fights after the initial setup.

5. **Adversarial reviews continued to catch real issues in every story.** Carried forward from Epics 1 and 2 — now proven effective across 3 consecutive epics.

6. **10 rich AI usage notes.** Epic 3 generated the most AI usage documentation of any epic — excellent raw material for the README's AI section (story 4.4).

## Challenges

1. **Story creation missed a planning gap and a visual gap.** AC #5 of story 3.1 ("Given the simulation is running...") had no task to satisfy it — the run loop was deferred to story 3.5, but the AC needed it in 3.1. Additionally, the story specified canvas colors but never considered what an empty grid looks like, resulting in an invisible canvas (background matching page background, no grid lines in spec).

2. **AI defaulted to quick fixes instead of architecturally correct solutions.** When hitting unexpected problems mid-implementation, the AI's instinct was "fix it fast and keep moving" rather than consulting the architecture or presenting options. In story 3.1, it proposed a basic rAF loop instead of the full §5.2 hook, and hardcoded `transpilePackages` instead of fixing auto-discovery. Both times the human had to redirect.

3. **AI did not visually verify UI output before marking stories ready for review.** The AI relied on programmatic checks (`curl`, `data-testid` assertions) rather than actually looking at the rendered page. The invisible canvas was caught by the human screenshotting the page.

4. **Invariant-test self-review still incomplete.** Stories 3.4 and 3.5 had missing or misleading tests caught in adversarial review (missing all-dead grid test, misleading keyboard test, missing rate-change verification test).

---

## Key Insights

1. **Planning gaps need belt-and-braces defense.** AC-to-task traceability should be verified at both story creation (the "belt") and story review before dev starts (the "braces"). For each AC, verify at least one task makes it satisfiable and testable. For UI stories, verify default/empty/error visual states are addressed.

2. **Unexpected problems that involve a choice between approaches are design decisions, not bug fixes.** When the dev agent encounters such a problem, it should stop implementing, document the problem, root cause, and candidate solutions, and wait for human input. The threshold: "does this require a choice between approaches?" If yes, document and ask. If it's a straightforward single-answer fix, proceed.

3. **Front-loading infrastructure is now a three-epic proven pattern.** Epics 1, 2, and 3 all validated the same strategy: the first story absorbs infrastructure cost, subsequent stories are progressively frictionless. This is a reliable strategy to continue applying.

4. **Forward-thinking architecture design eliminates rework.** The `genPerSecRef` pattern in `useSimulationLoop` was designed to anticipate story 3.5's needs. When 3.5 arrived, the hook needed zero changes. Investing in the right abstraction upfront saved an entire refactoring cycle.
